# Inference receipt — OBSERVE -> ENFORCE enablement

The gateway emits a **per-response inference receipt** for every `/v1/chat` turn: it fetches
the serving enclave's attestation (`GET https://api.redpill.ai/v1/attestation/report?model=<slug>`)
and surfaces `provider / model / app_id / compose_hash / no_log / signing_address / quote_present`
as the SSE `receipt` event. The receipt has a **2-phase rollout** (config + route already shipped
in `gateway/app/config.py` + `gateway/app/inference_backend.py`; this directory is the
data-collection + enablement side):

- **OBSERVE** (default): surface the receipt fields **without** enforcing an allowlist. The turn
  always completes; a flaky report endpoint is fail-soft. This is how we collect the real
  `app_id` / `compose_hash` set a `phala/` slug actually routes to before pinning anything.
- **ENFORCE** (auto-derived when an allowlist is set): the turn fails **CLOSED** — `event: error`,
  reply dropped, inference not even called — if the serving enclave's `app_id`/`compose_hash` is
  not in the allowlist **OR** `no_log` is not explicitly `true`.

Why observe first: **RedPill load-balances a single `phala/<model>` slug across multiple attested
providers** (we see `x-redpill-provider: phala` AND `near-ai`, plus a header-less `chutes` route —
`internal spike notes (d)`). Enforcing against one observation would fail-close the moment the
LB picks a sibling enclave. So we poll repeatedly, aggregate the distinct enclaves, and pin the
**union**.

## Contents

| File | What |
|---|---|
| `collect-app-ids.py` | Polls the report endpoint for each keep-warm slug, walks all three RedPill report schemas, aggregates distinct `(app_id, compose_hash)` enclaves, emits `observed-app-ids.json`. |
| `observed-app-ids.json` | The committed **live observation** + the **PROPOSED allowlist** (the two `GATEWAY_…` env values). Re-generated by the collector. Never contains the key. |

## How to run the collector

```sh
# Resolves the RedPill key from op://$OP_VAULT/redpill.ai/api_key (or $REDPILL_API_KEY).
# The key is NEVER printed, logged, or written to the output JSON.
python3 infra/receipt/collect-app-ids.py --rounds 8 --sleep 3
# -> writes infra/receipt/observed-app-ids.json
```

Flags: `--rounds N` (polls per slug), `--sleep S` (seconds between rounds), `--timeout S`
(per-request), `--models a,b,c` (override the default keep-warm set), `--out PATH`.

The default keep-warm set is the three slugs the gateway pings
(`gateway/app/config.py:keepwarm_models` / `swifty_inference/models.py`):
`phala/qwen3.6-35b-a3b-uncensored`, `phala/qwen3-30b-a3b-instruct-2507`, `phala/deepseek-v3.2`.

The near-ai **gateway-relay** enclave (`f550fdfb…`, `app_name: dstack-0.5.4`) is recorded in the

`(app_id, compose_hash)` (its stable identity), not `instance_id`.

### `no_log`: **NOT asserted by RedPill** (enforcement blocker)
RedPill's `/v1/attestation/report` does **not** expose a `no_log` field in **any** of the three
schemas. `parse_report` therefore records `no_log = unknown` for every enclave. Because the ENFORCE
gate fails closed unless `no_log` is **explicitly `true`**, flipping to enforce **today would reject
every turn** — even for the known-good qwen3.6 enclave. This is correct fail-closed behaviour, not a
bug; it is a **blocker on enforcement**, see below.

## How to flip OBSERVE -> ENFORCE

Enforcement is **a config change on the CVM only** — no code change. Setting **either** env flips
`gateway/app/config.py:receipt_enforce()` to `True`:

```sh
GATEWAY_INFERENCE_ALLOWED_APP_IDS=2c0a0c96cb6dbd659bf1446e2f3fce58172ff91b,9d31963336e6b0bdf8ba4c27b97442a818dd05fd
GATEWAY_INFERENCE_ALLOWED_COMPOSE_HASHES=537b750f6e99972db6d6ea24232ecf0c5826136f3e32704ad2936de75fca14a1,bcbccccf457a8f2dcd837b615ef56f63eaa707e3675f1ecd08ee81cb06ec2b28
```

(Copy the exact current values from `observed-app-ids.json -> proposed_allowlist`.) Set them in the
CVM's dstack compose env (`infra/cvm/dstack-compose.gw-v1.yml`) and redeploy.

### Blockers — do NOT enforce until these are resolved

1. **`no_log` is never asserted** → enforcing rejects 100% of turns. Resolve by either
   (a) getting RedPill to assert `no_log` in the report, or (b) making a **deliberate policy
   decision** to relax the `no_log` gate to a non-blocking observation. Do **not** silently drop a
   confidentiality claim — that is a claims-register matter (`docs/security/claims-register.md`).
2. **`phala/deepseek-v3.2` (chutes route) cannot be pinned** — its attestation carries no
   `app_id`/`compose_hash`, so an app_id/compose allowlist can never admit it. Either drop
   deepseek-v3.2 from the keep-warm set, route it via a pinnable provider, or extend enforcement to
   verify the `chutes` schema by a different identity (e.g. the quote measurement directly).
3. **Pin review before enforcing** — cross-check each `app_id`/`compose_hash` against the on-chain
   DstackApp registry + `docs/security/measurements.md` (`attestation-evidence.md §2.1, §4`: no
   runtime TOFU). The collector observes what RedPill *claims*; it does **not** verify the quote to
   the Intel/NVIDIA roots (that is the client verifier's job — `quote_verified_pending=True`).
4. **Re-run periodically** — RedPill may rotate enclaves / rebalance providers. A stale allowlist
   silently fails-closed on the new (legitimate) enclave. Re-run the collector and reconcile before
   each enforce push; consider a scheduled run.

## Trust note

This collector + allowlist is a **defence-in-depth** layer on the *gateway* side; the authoritative
trust decision is the **client** verifying the per-response quote to the Intel + NVIDIA roots and
checking the signer is bound into the quote (`contracts/attestation-evidence.md §3`). The gateway
allowlist catches a mid-session swap to an unexpected enclave early and keeps our *server* honest
about which enclaves it will relay to; it is not a substitute for client verification.
